============================
  PONG - GODOT CHEATSHEET
============================

--- CORE CONCEPTS ---

Scene:  A scene is a reusable building block in Godot. It's a tree of nodes
        saved as a .tscn file. Everything in your game is a scene — the main
        level, a paddle, a ball, a UI menu. Scenes can be nested inside other
        scenes (called "instancing").

Node:   The basic unit in Godot. Each node has a specific purpose (display a
        sprite, detect collisions, play audio, etc.). You build scenes by
        combining nodes in a tree hierarchy.

--- WHY NODE2D FOR THE MAIN SCENE? ---

Node2D is the base node for all 2D games. It gives you a 2D coordinate system
(x goes right, y goes down). We use it as the root of our main scene because
Pong is a 2D game and we just need a container to hold our child nodes (paddles,
ball, walls, score label). It doesn't do much on its own — it's an organizer.

--- AUTOLOADS ---

GameSettings (scenes/global/game_settings.gd)
  An Autoload singleton available everywhere as GameSettings.
  Stores runtime settings that persist across scene changes:
    ai_enabled            → bool: whether the current match is PvE
    ai_difficulty         → int: 0=easy, 1=medium, 2=hard
    fullscreen            → bool
    win_score             → int (default 3, range 1–15)
    ball_speed            → float (default 400, range 200–800)
    master_volume         → float 0.0–1.0
    last_input_is_controller → bool: tracks keyboard vs gamepad input
  Methods:
    apply_fullscreen()    → Switches between windowed and fullscreen
    apply_volume()        → Converts 0–1 volume to dB on Master bus

UiSounds (scenes/global/ui_sounds.gd)
  An Autoload singleton that auto-connects to every BaseButton in the tree.
  Plays a hover sound whenever a button receives mouse_entered or focus_entered.
  Uses get_tree().node_added to catch buttons added at runtime.

--- COLLISION LAYERS ---

Layer 1: Walls (StaticBody2D)      → TopWall, BottomWall
Layer 2: Paddles (CharacterBody2D) → LeftPaddle, RightPaddle
Layer 3: Ball (CharacterBody2D)    → Ball

Ball is on layer 3, masks 1+2      → collides with walls and paddles
Paddle is on layer 2, masks 1+3    → collides with walls and ball

--- NODE TYPES USED ---

Node2D               - Base 2D node, used as a container/root
StaticBody2D          - A physics body that does NOT move (walls)
CharacterBody2D       - A physics body you move with code (paddles, ball)
CollisionShape2D      - Defines the hitbox shape; child of a physics body
ColorRect             - A simple colored rectangle (visuals for paddle/ball/center line)
Label                 - Displays text on screen (score, titles, winner)
Button               - A clickable button, emits pressed() signal
CheckButton          - A toggle button (on/off), emits toggled(bool) signal
HSlider              - A horizontal slider, emits value_changed(float) signal
Control              - Base node for all UI elements. Supports anchors/layout.
VBoxContainer        - Arranges children vertically with spacing
HBoxContainer        - Arranges children horizontally with spacing
CanvasLayer          - Renders children on a separate layer (used for UI on
                       top of Node2D scenes — required because Control anchors
                       don't work under a Node2D parent)
AudioStreamPlayer     - Plays a sound globally (not 2D-positioned) — victory sound
AudioStreamPlayer2D   - Plays a sound in 2D space (paddle hit, wall hit, scored)

--- PHYSICS BODIES CHEAT SHEET ---

StaticBody2D      → Doesn't move. Other things bounce off it. (walls)
CharacterBody2D   → You control movement in code. (paddle, ball)
RigidBody2D       → Physics engine moves it (gravity, forces). Not used here.
Area2D            → Detects overlaps but doesn't collide. (triggers, pickups)

--- INPUT MAP ---

All actions defined in Project Settings > Input Map. Each action maps
keyboard + controller inputs so both work automatically.

  ui_accept    → Enter, Space, Gamepad A (South)
  ui_cancel    → Escape, Gamepad B (East), Gamepad Start
  ui_restart   → R, Gamepad Y (North)

  p1_up        → W, Left Stick Up, D-pad Up
  p1_down      → S, Left Stick Down, D-pad Down
  p2_up        → Up Arrow, Right Stick Up
  p2_down      → Down Arrow, Right Stick Down

In PvE mode, the left paddle also listens to p2_up/p2_down as alt actions
so the player can use either set of controls.

--- SIGNALS IN THIS PROJECT ---

signal scored(player: String)           → Declared in ball.gd
  scored.emit("left")                   → Ball emits when it exits right side
  scored.emit("right")                  → Ball emits when it exits left side
  ball.scored.connect(_on_ball_scored)   → game.gd listens for it in _ready()

signal paddle_hit(paddle: CharacterBody2D)  → Declared in ball.gd
  paddle_hit.emit(collider)                 → Ball emits on paddle collision
  ball.paddle_hit.connect(_on_ball_paddle_hit) → game.gd listens, re-rolls AI error

Signals are Godot's event system. A node emits a signal, other nodes listen.
This keeps code decoupled — the ball doesn't need to know about the score.

--- AI SYSTEM ---

PaddleAI (class_name, static methods in paddle_ai.gd):
  Three difficulty strategies, each returns a motion Vector2:

  Easy   - Chases ball's Y position directly. Tracks slower when ball moves
           away. Has a reaction delay (0.15s) when ball first approaches.
           Error range: ±65px.

  Medium - Predicts straight-line shots using BallPredictor (no bounces).
           Falls back to chasing if wall bounce would occur. Also randomly
           uses chase instead of prediction 20% of the time. Drifts to
           center when ball moves away. Error range: ±30px.

  Hard   - Predicts with wall bounces (up to 10). Moves 20% faster.
           Returns to center promptly when ball heads away.
           Error range: ±15px.

  Error offset: randomized each time the ball hits the AI paddle
  (via randomize_ai_error()). Adds imprecision so the AI isn't perfect.

BallPredictor (class_name, static methods in ball_predictor.gd):
  predict_y()       → Returns the Y coordinate where ball reaches target_x.
                       Set include_bounces=false for straight-line only.
  simulate_path()   → Returns Array[Vector2] of points along the ball's
                       predicted path, bouncing off top/bottom walls.
  draw_prediction() → Draws yellow path lines + red endpoint (debug overlay).
                       Called from ball._draw() when debug_draw=true.

--- PADDLE CONFIGURATION ---

Paddles are configured at runtime via paddle.configure():
  configure(up_action, down_action, ball?, alt_up?, alt_down?)

  PvP mode:
    left_paddle.configure("p1_up", "p1_down")
    right_paddle.configure("p2_up", "p2_down")

  PvE mode:
    left_paddle.configure("p1_up", "p1_down", null, "p2_up", "p2_down")
    right_paddle.configure("p2_up", "p2_down", ball)  ← AI-controlled

  When a ball reference is passed, the paddle becomes AI-controlled.
  The paddle has separate speeds: player_speed (600) and speed (400, AI base).

--- BALL BOUNCE MECHANICS ---

Angle-based paddle bounce (not simple reflection):
  - Where the ball hits the paddle is normalized to -1 (top) to +1 (bottom)
  - This offset is multiplied by max_bounce_angle (50°) to get the angle
  - Hitting center → flat horizontal bounce (0°)
  - Hitting edge → steepest possible angle (50°)
  - Horizontal direction is always reversed on paddle hit
  - Speed increases by speed_increment (×1.05) on every hit, capped at max_speed

--- DYNAMIC LAYOUT ---

game.gd._layout_game_elements() positions all game objects based on
the current viewport size, replacing hardcoded pixel positions:
  - Walls span full viewport width, placed just outside visible area
  - Paddles centered vertically, offset PADDLE_MARGIN (40px) from edges
  - Score label spans full width at top
  - Center line at viewport midpoint

--- GDSCRIPT BASICS ---

extends Node2D        → This script inherits from Node2D
func _ready():        → Called once when the node enters the scene
func _process(delta): → Called every frame (for visuals/input)
func _physics_process(delta): → Called every physics tick (for movement)
func _unhandled_input(event):  → Called for input not consumed by UI
func _input(event):            → Called for ALL input events (before UI)
func _draw():                  → Custom drawing (called after queue_redraw())
@export var speed := 400.0    → Editable in the Inspector panel
@onready var ball = $Ball     → Gets a child node reference when ready
await <signal>                → Pause execution until a signal fires
class_name MyName             → Registers a global name for this script
                                (accessible from any other script without
                                preloading — great for constants/enums)
static func name()            → A function that belongs to the class, not an
                                instance. Called as ClassName.name().

--- NODE REFERENCES ---

$NodeName              → Get a direct child by name
$Parent/Child          → Get a nested child by path
%UniqueName            → Get a node marked as "Unique Name" in the scene tree.
                         Works regardless of where the node is in the hierarchy.
                         Set via: right-click node → Access as Unique Name.
@onready var x = %Name → Combine with @onready for typed, path-independent refs

--- PROCESS MODES ---

Every node has a Process Mode (Inspector > Node > Process > Mode):
  Inherit        → Uses parent's mode (default)
  Pausable       → Stops when get_tree().paused = true
  When Paused    → ONLY runs when paused (opposite of Pausable)
  Always         → Runs regardless of pause state
  Disabled       → Never processes

The pause menu uses "Always" so it can receive Escape input both
when the game is running (to pause) and when paused (to resume).

--- SCENE MANAGEMENT ---

change_scene_to_file(path)  → Replaces the entire current scene with a new one.
                              Old scene is freed. Used for menu navigation.
reload_current_scene()      → Reloads the current scene from scratch (restart).
CanvasLayer + instancing    → For overlays (like pause menu) that should appear
                              ON TOP of the game without replacing it.

Routes (class_name)         → Central file with scene path constants so paths
                              are defined in one place. Usage: Routes.MAIN_MENU
                              NOT an Autoload — no instance created, just a
                              globally available class for static constants.
                              Constants: MAIN_MENU, DIFFICULTY_MENU,
                              SETTINGS_MENU, PLACEHOLDER_SCREEN, GAME

--- UI / CONTROL NODES ---

Control nodes use anchors to position themselves relative to their parent.
Anchors only work when the parent is also a Control (or a CanvasLayer).

Anchors Preset (set via Layout button at top of 2D viewport):
  Full Rect    → Stretches to fill the entire parent
  Center       → Sits at the center of the parent
  Tip: Hold Shift when clicking a preset to also reset offsets.

UI under Node2D: Control anchors DON'T WORK under a Node2D parent.
  Solution: Add a CanvasLayer between the Node2D and the Control nodes.

StyleBox resources (.tres): Saved style overrides that can be shared across
  multiple buttons/controls. Edit once, all linked nodes update.

visible property: Built-in on all CanvasItem nodes (Control, Node2D, etc.).
  show() sets visible=true, hide() sets visible=false.
  Used to toggle UI elements like the pause menu.

grab_focus(): Gives keyboard/controller focus to a Control node. Used on
  the first button in each menu so controller navigation works immediately.

--- CONTROLLER SUPPORT ---

Input actions map both keyboard keys and gamepad buttons/sticks.
GameSettings.last_input_is_controller tracks the most recent input device
so the game can show context-appropriate button prompts:
  Keyboard → "Press R to Restart · Esc for Menu"
  Gamepad  → "Press △ to Restart · ○ for Menu"
The restart label updates every frame via _process() to stay in sync.

All menus call grab_focus() on the first button so d-pad/stick navigation
works without needing a mouse click first.

--- COORDINATE SYSTEM ---

Origin (0,0) is the TOP-LEFT of the screen.
X increases to the RIGHT.
Y increases DOWNWARD.
Our viewport is 1152 x 648 pixels.

So:
  Top-left corner     = (0, 0)
  Top-right corner    = (1152, 0)
  Bottom-left corner  = (0, 648)
  Center              = (576, 324)

--- GAME FLOW ---

1. Game launches → Main Menu appears
2. Player picks Play Co-op → game.tscn loads (PvP mode)
3. Player picks Play AI → Difficulty Menu → pick Easy/Medium/Hard → game.tscn loads (PvE mode)
4. Player picks Settings → Settings Menu (fullscreen, win score, ball speed, volume)
5. game.gd _ready() connects ball's scored and paddle_hit signals
6. game.gd _ready() calls configure() on paddles (PvP or PvE based on GameSettings)
7. game.gd _layout_game_elements() positions walls/paddles/UI based on viewport
8. Ball resets to center, waits 1 second, launches in random direction
9. Players move paddles with W/S and Up/Down (or gamepad sticks/d-pad)
10. In PvE, right paddle is AI-controlled; left paddle accepts both input sets
11. Ball bounces off walls (reflection) and paddles (angle-based), speeding up each hit
12. When ball exits left/right edge → scored signal → score updates
13. Ball resets to center and re-launches
14. First player to win_score points wins → game freezes, winner label + victory sound
15. Press R (or gamepad Y) to restart
16. Press Escape (or gamepad B/Start) → pause menu (resume, restart, or main menu)
17. Quit from main menu → closes the game

--- EDITOR TIPS ---

Renaming files:  Always rename via the FileSystem dock (right-click → Rename).
                 Godot auto-updates all references. Renaming outside Godot
                 breaks scene/script references.

Re-attaching scripts: If a script reference breaks, select the node →
                      Inspector → Script property → drag the file from
                      FileSystem dock. No need to detach/reattach.
